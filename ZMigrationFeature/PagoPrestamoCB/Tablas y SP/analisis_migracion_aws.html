Análisis de Migración a AWS - Sistema de Corresponsales Bancarios

1. Análisis de Datos Actuales

1.1 Tabla MW_CORRESPONSALES_BANCARIOS

Estructura:

  • CODIGO_CORRESPONSAL (PK)
  • NOMBRE_CORRESPONSAL
  • ESTADO
  • CUENTA_CREDITO
  • CUENTA_DEBITO
  • MONEDA_PERMITIDA
  • CODIGO_PAIS
  • CUENTA_COMISION

Características: Tabla de configuración con 3 registros. Datos maestros de baja frecuencia de cambio.

1.2 Tabla MW_TRANSACCIONES_CORBAN

Estructura:

  • CODIGO_CORRESPONSAL (FK)
  • CODIGO_TRANSACCION
  • DESCRIPCION
  • ESTADO
  • MONTO_MAX_X_TRN
  • TIPO_TRANSACCION
  • CODIGO_PAIS
  • MONTO_COMISION
  • MONEDA_COMISION
  • TIPO_TRANSACCION_COMISION

Características: Tabla de configuración con 27 registros. Define tipos de transacciones permitidas por corresponsal.

1.3 Stored Procedure MW_P_VALIDA_CORBAN

Funcionalidad:

  • Valida existencia y estado del corresponsal bancario
  • Recupera configuración de cuentas (crédito, débito, comisión)
  • Valida tipo de transacción permitida
  • Retorna información de comisiones y monedas
  • Manejo de errores con códigos y mensajes descriptivos

Patrón: Consulta de validación y configuración (READ-ONLY), sin lógica de negocio compleja.

3. Mejor Opción Recomendada: Amazon DynamoDB

Justificación:

  • Naturaleza de los datos: Son tablas de configuración pequeñas (30 registros totales) con baja frecuencia de cambios
  • Patrón de acceso: Consultas simples por clave primaria (CODIGO_CORRESPONSAL + CODIGO_PAIS + CODIGO_TRANSACCION)
  • Escalabilidad: DynamoDB escala automáticamente sin intervención
  • Costo: Para este volumen de datos, el costo mensual sería < $5 USD
  • Disponibilidad: 99.99% SLA sin configuración adicional
  • Latencia: Respuestas en milisegundos, ideal para validaciones en tiempo real
  • Mantenimiento: Cero administración de infraestructura
  • Multi-región: Global Tables permite replicación automática entre regiones

Diseño de Tablas en DynamoDB:

Tabla 1: CorresponsalesBancarios

  • Partition Key: CODIGO_CORRESPONSAL#CODIGO_PAIS (ej: "101#HN01")
  • Atributos: nombre_corresponsal, estado, cuenta_credito, cuenta_debito, moneda_permitida, cuenta_comision

Tabla 2: TransaccionesCorban

  • Partition Key: CODIGO_CORRESPONSAL#CODIGO_PAIS (ej: "101#HN01")
  • Sort Key: CODIGO_TRANSACCION (ej: "2")
  • Atributos: descripcion, estado, monto_max_x_trn, tipo_transaccion, monto_comision, moneda_comision, tipo_transaccion_comision

Alternativa Híbrida (Recomendada para Producción):

DynamoDB + ElastiCache (Redis)

  • DynamoDB como fuente de verdad persistente
  • Redis como caché de primer nivel para latencia ultra-baja
  • TTL de 1 hora en caché (datos cambian raramente)
  • Costo adicional mínimo (~$15-20/mes para instancia t4g.micro)
  • Latencia < 1ms en el 99% de consultas

5. Arquitectura Recomendada Final

Opción Óptima: API REST Serverless con Caché

Componentes:

  1. Amazon API Gateway (REST API)
    • Endpoint: POST /api/v1/corresponsales/validar
    • Autenticación: API Key + IAM
    • Throttling: 1000 req/seg
    • Caché de API Gateway: 5 minutos (opcional)
  2. AWS Lambda (Python 3.12 o Node.js 20)
    • Función: ValidarCorresponsalFunction
    • Memoria: 256 MB
    • Timeout: 10 segundos
    • Lógica: Réplica del SP actual
  3. Amazon ElastiCache (Redis) - Capa 1
    • Instancia: cache.t4g.micro
    • TTL: 3600 segundos (1 hora)
    • Patrón: Cache-Aside
  4. Amazon DynamoDB - Capa 2
    • Modo: On-Demand (pago por request)
    • Tablas: CorresponsalesBancarios, TransaccionesCorban
    • Backup: Point-in-time recovery habilitado
  5. Amazon CloudWatch
    • Logs de Lambda
    • Métricas personalizadas
    • Alarmas de errores
  6. AWS X-Ray
    • Trazabilidad end-to-end
    • Análisis de latencia

Flujo de Ejecución:

  1. Cliente invoca API Gateway con parámetros (codigo_corresponsal, codigo_pais, codigo_transaccion)
  2. API Gateway valida autenticación y throttling
  3. Lambda recibe request y genera cache key
  4. Lambda consulta Redis (caché L1)
    • Si existe: retorna inmediatamente (latencia ~1ms)
    • Si no existe: continúa al paso 5
  5. Lambda consulta DynamoDB (caché L2)
    • GetItem en CorresponsalesBancarios
    • GetItem en TransaccionesCorban
  6. Lambda valida estado y construye respuesta
  7. Lambda almacena resultado en Redis
  8. Lambda retorna respuesta a API Gateway
  9. API Gateway retorna al cliente

7. Plan de Migración Recomendado

Fase 1: Preparación (1 semana)

  • Crear tablas en DynamoDB
  • Migrar datos desde Oracle usando script Python/boto3
  • Configurar ElastiCache Redis
  • Validar integridad de datos

Fase 2: Desarrollo (2 semanas)

  • Desarrollar función Lambda con lógica del SP
  • Implementar caché Redis
  • Crear API Gateway con endpoints
  • Desarrollar tests unitarios e integración
  • Configurar CloudWatch y X-Ray

Fase 3: Testing (1 semana)

  • Pruebas funcionales
  • Pruebas de carga (JMeter/Gatling)
  • Validación de latencia
  • Pruebas de failover

Fase 4: Despliegue (1 semana)

  • Despliegue en ambiente de staging
  • Pruebas de aceptación
  • Despliegue en producción (Blue/Green)
  • Monitoreo intensivo 24/7
  • Rollback plan preparado

8. Ventajas de la Arquitectura Propuesta

  • ✅ Serverless: Cero gestión de servidores
  • ✅ Auto-escalable: Maneja de 1 a 1M requests sin cambios
  • ✅ Alta disponibilidad: Multi-AZ por defecto
  • ✅ Bajo costo: ~$23/mes vs $200+/mes con RDS
  • ✅ Baja latencia: < 10ms con caché, < 50ms sin caché
  • ✅ Fácil mantenimiento: Código versionado en Git
  • ✅ Testing: Tests unitarios y de integración simples
  • ✅ Observabilidad: Logs, métricas y trazas completas
  • ✅ Seguridad: IAM, API Keys, VPC (opcional)
  • ✅ Multi-región: Fácil replicación con Global Tables

9. Consideraciones Adicionales

Seguridad:

  • Implementar AWS WAF en API Gateway para protección DDoS
  • Usar AWS Secrets Manager para credenciales
  • Habilitar encryption at rest en DynamoDB
  • Usar VPC endpoints para comunicación privada

Monitoreo:

  • Alarmas CloudWatch para errores > 1%
  • Alarmas para latencia p99 > 100ms
  • Dashboard con métricas clave
  • SNS para notificaciones a equipo

Disaster Recovery:

  • Backups automáticos de DynamoDB (PITR)
  • Replicación cross-region con Global Tables
  • Versionamiento de código Lambda
  • Infraestructura como código (Terraform/CloudFormation)

10. Conclusión

La migración de las tablas de corresponsales bancarios a Amazon DynamoDB combinado con ElastiCache Redis y la transformación del stored procedure a una API REST serverless con Lambda representa la mejor opción para este caso de uso.

Esta arquitectura ofrece el mejor balance entre costo, rendimiento, escalabilidad y mantenibilidad, con un costo operativo aproximado de $23 USD/mes y latencias inferiores a 10ms en el 99% de las consultas.

El tiempo estimado de migración completa es de 5 semanas con un equipo de 2 desarrolladores, y el ROI se alcanza en menos de 3 meses comparado con mantener infraestructura tradicional.
